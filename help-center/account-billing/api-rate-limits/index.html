api rate limits

understand the rate limits applied to your organization's api usage.

limits at a glance

rate limits are applied per endpoint type and reset independently.

endpoint limit window applies to
/v1/score 30 requests per 15 minutes (rolling) authenticated api key
/v1/public/score 20 requests per day (resets midnight UTC) ip address
/v1/public/score 3 requests lifetime no reset browser fingerprint

authenticated api key — /v1/score

requests to /v1/score authenticated with an x-api-key header are subject to a rolling window of 30 requests per 15 minutes. the window is per organization — all team members sharing the same api key draw from the same pool. when the limit is exceeded, the api returns 429 too many requests until the oldest requests in the window fall outside the 15-minute boundary.

public endpoint — /v1/public/score

the public endpoint allows unauthenticated scans for evaluation and prototyping. it is enforced by two independent limits that both apply simultaneously:

  • 20 scans per day per ip address. the counter resets at midnight UTC.
  • 3 scans lifetime per browser fingerprint. this limit does not reset. it exists to prevent automated abuse of the public tier.

the public endpoint is not intended for production workloads. it does not consume credits and does not require authentication, but it has no sla or consistency guarantees.

handling 429 responses

when your api key hits the rate limit, the api returns 429 too many requests. the response body includes a retryAfter field indicating the number of seconds until the limit resets. implement exponential backoff in your integration and respect this value rather than polling immediately:

  • read the retryAfter value from the response body.
  • wait at least that many seconds before retrying.
  • if you are consistently hitting the limit, consider batching scans or contacting us about a higher-volume plan.
